# Story 00.03 - Auth Store

> Epic: 0 - Foundation
> Depends on: 00.02
> Backend contract source: `server/SERVER_CONTEXT.md`

## Story Header

**As a** QC member,
**I want** my authenticated session held securely in the browser,
**so that** I can use protected dashboard screens without exposing JWTs to persistent storage.

## In Scope

- [WEB] Zustand auth store with memory-only access token.
- [WEB] Store current `UserResponse`.
- [WEB] Auth helper hook.

## Out of Scope

- Server-side session middleware.
- OAuth persistence.

## Deferred

- OAuth login UX is deferred because backend OAuth linking is incomplete.

## API/Data Contract

```ts
type UserResponse = {
  publicId: string;
  email: string;
  displayName: string;
  role: 'QC_MEMBER' | 'QC_LEAD' | 'ADMIN';
  status: 'PENDING_EMAIL_VERIFICATION' | 'ACTIVE' | 'DISABLED';
  lastLoginAt: string | null;
};
```

## UI States

- `isAuthenticated` is true only when an access token exists and user is loaded.
- On page refresh, the app starts unauthenticated until refresh/bootstrap succeeds.

## Acceptance Criteria

### AC 1 - Login state is memory-only

```text
Platform: [WEB]
Actor: QC member

Given  the login API returns an access token and user
When   the app stores the session
Then   the token exists only in Zustand memory
And    localStorage does not contain the JWT
```

### AC 2 - Logout clears session

```text
Platform: [WEB]
Actor: QC member

Given  the auth store has an access token and user
When   logout runs
Then   accessToken and user are cleared
And    isAuthenticated becomes false
```

## Implementation Checklist

- Create `src/lib/store/auth-store.ts`.
- Create `src/hooks/use-auth.ts`.
- Provide `login`, `logout`, `setToken`, and `setUser`.
- Keep refresh-token behavior in API client, not in localStorage.

## Verification Notes

- Manually verify browser storage after login.
