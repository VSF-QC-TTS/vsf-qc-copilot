# E2E Runbook: QC Evaluation Flow

Date: 2026-06-15

Purpose: run one complete VSF QC Copilot flow from account setup to evaluation results and export. This document is the source of truth for manual E2E checks while the automated `scripts/e2e-smoke.sh` is being updated to the latest rubric/judge-model contracts.

## Scope

This runbook covers:

1. Start local services.
2. Register and activate a QC user.
3. Create a project.
4. Create a target API connector to the local mock chatbot.
5. Create and approve a dataset with test cases.
6. Create a reusable rubric with structured criteria and publish v1.
7. Create a project judge model.
8. Start an evaluation run.
9. Poll job and inspect results/events.
10. Add a QC review decision.
11. Create and download an export.

## Execution Modes

Use one mode per run:

| Mode | Use when | Notes |
| --- | --- | --- |
| `mock` | Fast product workflow smoke test | Default in `application-dev.yaml`. Does not execute Promptfoo CLI. Result status/score are generated by `MockPromptfooExecutor`. |
| `cli` | Real Promptfoo rubric/judge integration test | Requires local Promptfoo binary and a real judge provider API key. Current `server/Dockerfile` runtime does not install Promptfoo, so prefer running the server natively for this mode until the image is updated. |

## API Smoke Script

Run these commands from repo root. They intentionally use explicit API calls so failures show the broken boundary.

Set helpers:

```bash
export BASE_URL=http://localhost:8080
export EMAIL="e2e-$(date +%s)@test.local"
export PASSWORD='TestPass123!'

api() {
  local method="$1"
  local path="$2"
  shift 2
  curl -sS -X "$method" \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer ${ACCESS_TOKEN:-}" \
    "$@" \
    "$BASE_URL$path"
}
```

### 1. Register, Activate, Login

```bash
REGISTER_RESPONSE=$(api POST /api/v1/auth/register -d "{
  \"email\": \"$EMAIL\",
  \"password\": \"$PASSWORD\",
  \"displayName\": \"E2E QC User\"
}")
echo "$REGISTER_RESPONSE" | jq .

docker compose exec -T db psql -U vqc -d vqc_dev -c \
  "UPDATE users SET status = 'ACTIVE' WHERE username = '$EMAIL';"

LOGIN_RESPONSE=$(api POST /api/v1/auth/login -d "{
  \"email\": \"$EMAIL\",
  \"password\": \"$PASSWORD\"
}")
export ACCESS_TOKEN=$(echo "$LOGIN_RESPONSE" | jq -r '.accessToken')
test "$ACCESS_TOKEN" != "null"
```

Expected:

- Register returns a user `publicId`.
- Login returns `accessToken`.

### 2. Create Project

```bash
PROJECT_RESPONSE=$(api POST /api/v1/projects -d '{
  "name": "E2E QC Project",
  "description": "End-to-end evaluation runbook project"
}')
export PROJECT_ID=$(echo "$PROJECT_RESPONSE" | jq -r '.publicId')
echo "$PROJECT_RESPONSE" | jq .
```

Expected:

- `PROJECT_ID` is a UUID.

### 4. Create Dataset And Test Cases

```bash
DATASET_RESPONSE=$(api POST "/api/v1/projects/$PROJECT_ID/datasets" -d '{

  "groundTruth": "Today you walked 8,200 steps.",

api POST "/api/v1/datasets/$DATASET_ID/test-cases" -d '{
```

Approve dataset:

```bash

```

Expected:

- Dataset status becomes `APPROVED`.
- Approval fails if there are zero active test cases.

### 5. Create And Publish Rubric

Create a user-scoped rubric with draft v1 and criteria in one request:

```bash
RUBRIC_VERSION_RESPONSE=$(api POST /api/v1/rubrics -d '{
  "name": "E2E Semantic Answer Rubric",
  "description": "Checks whether chatbot answers match expected answers semantically.",
  "content": "Judge the chatbot answer against the user question, expected answer, preconditions, and metadata. Use semantic equivalence; do not require exact wording unless a criterion explicitly requires it.",
  "outputSchemaJson": "{\"type\":\"object\",\"required\":[\"final_status\",\"scores\",\"reason\"],\"properties\":{\"final_status\":{\"enum\":[\"PASS\",\"FAIL\",\"WARNING\"]},\"scores\":{\"type\":\"object\"},\"reason\":{\"type\":\"string\"}}}",
  "criteria": [
    {
      "name": "Expected Answer Match",
      "description": "The actual answer should preserve the core facts of the expected answer.",
      "weight": 60,
      "passCondition": "The answer is semantically consistent with the expected answer and testcase context.",
      "failCondition": "The answer contradicts, fabricates, or omits the expected core answer.",
      "judgeInstruction": "Compare the actual output with {{groundTruth}} for the question {{question}}. Consider {{precondition}} and {{metadata}} as context.",
      "metricKey": "expected_answer_match",
      "isCritical": true,
      "sortOrder": 1
    },
    {
      "name": "Clarity",
      "description": "The answer should be clear enough for QC review.",
      "weight": 40,
      "passCondition": "The answer is understandable and directly addresses the user.",
      "failCondition": "The answer is vague, confusing, or unrelated.",
      "judgeInstruction": "Assess whether the actual output is clear, concise, and directly responsive to {{question}}.",
      "metricKey": "clarity",
      "isCritical": false,
      "sortOrder": 2
    }
  ]
}')
export RUBRIC_VERSION_ID=$(echo "$RUBRIC_VERSION_RESPONSE" | jq -r '.publicId')
export RUBRIC_ID=$(echo "$RUBRIC_VERSION_RESPONSE" | jq -r '.rubricPublicId')
echo "$RUBRIC_VERSION_RESPONSE" | jq .
```

Publish v1:

```bash
api PATCH "/api/v1/rubric-versions/$RUBRIC_VERSION_ID" -d '{
  "status": "PUBLISHED"
}' | jq .
```

Expected:

- Create response is a `RubricVersionResponse`.
- `RUBRIC_VERSION_ID` is a UUID.
- Publish response has `status: "PUBLISHED"`.
- Published version has nonblank `content` and at least one criterion.

### 6. Create Judge Model

For `mock` mode, the API key can be a non-empty placeholder because the mock executor does not call the provider. For `cli` mode, use a real key.

```bash
export JUDGE_API_KEY="${GEMINI_API_KEY:-test-gemini-key}"

JUDGE_RESPONSE=$(api POST "/api/v1/projects/$PROJECT_ID/judge-models" -d "{
  \"name\": \"E2E Gemini Judge\",
  \"provider\": \"GEMINI\",
  \"modelName\": \"gemini-2.5-flash\",
  \"apiKey\": \"$JUDGE_API_KEY\",
  \"configJson\": null,
  \"active\": true
}")
export JUDGE_MODEL_ID=$(echo "$JUDGE_RESPONSE" | jq -r '.publicId')
echo "$JUDGE_RESPONSE" | jq .
```

Expected:

- Response includes `apiKeyMasked`, not the raw key.
- `active` is `true`.

### 7. Start Evaluation Run

```bash
RUN_RESPONSE=$(api POST "/api/v1/projects/$PROJECT_ID/evaluation-runs" -d "{
  \"datasetPublicId\": \"$DATASET_ID\",
  \"rubricVersionPublicId\": \"$RUBRIC_VERSION_ID\",
  \"targetConnectorPublicId\": \"$CONNECTOR_ID\",
  \"judgeModelPublicId\": \"$JUDGE_MODEL_ID\",
  \"maxConcurrency\": 1
}")
export RUN_ID=$(echo "$RUN_RESPONSE" | jq -r '.runPublicId')
export JOB_ID=$(echo "$RUN_RESPONSE" | jq -r '.jobPublicId')
echo "$RUN_RESPONSE" | jq .
```

Expected:

- HTTP status is `202`.
- `status` is `PENDING`.
- Both `RUN_ID` and `JOB_ID` are UUIDs.

### 8. Poll Job

```bash
for i in $(seq 1 60); do
  JOB_RESPONSE=$(api GET "/api/v1/jobs/$JOB_ID")
  JOB_STATUS=$(echo "$JOB_RESPONSE" | jq -r '.status')
  echo "$JOB_RESPONSE" | jq '{status, progressCurrent, progressTotal, errorMessage}'

  if [ "$JOB_STATUS" = "COMPLETED" ]; then
    break
  fi
  if [ "$JOB_STATUS" = "FAILED" ] || [ "$JOB_STATUS" = "CANCELLED" ]; then
    echo "$JOB_RESPONSE" | jq .
    exit 1
  fi
  sleep 3
done
```

Expected:

- Job moves through `PENDING`/`RUNNING` to `COMPLETED`.
- If it fails, inspect `errorMessage` and `docker compose logs api`.

### 9. Inspect Run, Events, And Results

```bash
api GET "/api/v1/evaluation-runs/$RUN_ID" | jq .
api GET "/api/v1/evaluation-runs/$RUN_ID/events" | jq .

RESULTS_RESPONSE=$(api GET "/api/v1/evaluation-runs/$RUN_ID/results?page=0&size=20")
echo "$RESULTS_RESPONSE" | jq .
export FIRST_RESULT_ID=$(echo "$RESULTS_RESPONSE" | jq -r '.items[0].publicId')
```

Expected in `mock` mode:

- Results exist for all active test cases.
- `rawPromptfooResultJson` contains `{"mode":"mock"}` if exposed by the response shape.
- Status may be random because the mock executor intentionally randomizes `PASS`/`FAIL`/`WARNING`.

Expected in `cli` mode:

- `criteriaResultsJson` is present for rubric assertions.
- Any failed criterion forces result `judgeStatus: "FAIL"`.
- Unknown/non-rubric metrics are ignored in weighted score.
- `runs/{RUN_ID}/tests.json` includes `groundTruth` in `vars`.
- `runs/{RUN_ID}/tests.json` uses `llm-rubric` assertions instead of adding exact `contains` when criteria exist.

### 10. Add QC Review Decision

```bash
api PUT "/api/v1/evaluation-results/$FIRST_RESULT_ID/review-decision" -d '{
  "qcStatus": "PASS",
  "qcNote": "E2E review completed. Result accepted.",
  "picBugUserPublicId": null
}' | jq .

api GET "/api/v1/evaluation-results/$FIRST_RESULT_ID/review-decision" | jq .
```

Expected:

- Review response has `qcStatus: "PASS"`.
- `NOT_REVIEWED` must not be accepted as a write status.

### 11. Create And Download Export

```bash
EXPORT_RESPONSE=$(api POST "/api/v1/evaluation-runs/$RUN_ID/exports" -d '{
  "fileType": "EXCEL"
}')
export EXPORT_ID=$(echo "$EXPORT_RESPONSE" | jq -r '.exportPublicId')
export EXPORT_JOB_ID=$(echo "$EXPORT_RESPONSE" | jq -r '.jobPublicId')
echo "$EXPORT_RESPONSE" | jq .

for i in $(seq 1 60); do
  EXPORT_JOB_RESPONSE=$(api GET "/api/v1/jobs/$EXPORT_JOB_ID")
  EXPORT_JOB_STATUS=$(echo "$EXPORT_JOB_RESPONSE" | jq -r '.status')
  echo "$EXPORT_JOB_RESPONSE" | jq '{status, progressCurrent, progressTotal, errorMessage}'

  if [ "$EXPORT_JOB_STATUS" = "COMPLETED" ]; then
    break
  fi
  if [ "$EXPORT_JOB_STATUS" = "FAILED" ] || [ "$EXPORT_JOB_STATUS" = "CANCELLED" ]; then
    echo "$EXPORT_JOB_RESPONSE" | jq .
    exit 1
  fi
  sleep 3
done

curl -sS \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -o e2e-export.xlsx \
  "$BASE_URL/api/v1/exports/$EXPORT_ID/file"

ls -lh e2e-export.xlsx
```

Expected:

- Export job completes.
- `e2e-export.xlsx` exists and is larger than a tiny JSON error response.

## Manual UI Scenario

Use this after API smoke is green.

1. Open `http://localhost:3000/vi/login`.
2. Login with the E2E user.
3. Create or open `E2E QC Project`.
4. Add target connector:
   - URL: `http://localhost:8080/mock-chatbot/chat`
   - Method: `POST`
   - Body: `{"message":"{{question}}"}`
   - Response selector: `$.answer`
5. Run connector test.
6. Create dataset and two test cases.
7. Approve dataset.
8. Create rubric through the rubric wizard:
   - Confirm preview content is editable.
   - Confirm criteria metric keys are lowercase snake_case.
   - Publish v1.
9. Create judge model under the project.
10. Start evaluation:
    - Select approved dataset.
    - Select active connector.
    - Select published rubric version.
    - Select active judge model.
11. Open run detail:
    - Watch job events.
    - Inspect result table.
    - Open one result and set QC decision.
12. Export results as Excel and download the file.

## Pass Criteria

The E2E run passes when all of these are true:

- User can authenticate.
- Project, connector, dataset, rubric, judge model, evaluation run, review decision, and export are created through public APIs/UI.
- Dataset approval enforces active test cases.
- Evaluation run returns a job and eventually completes.
- Evaluation results are visible and tied to original test cases.
- QC review decision can be written and read back.
- Export file can be downloaded.
- In `cli` mode, generated Promptfoo files use rubric `llm-rubric` assertions and `groundTruth` as judge context.

## Troubleshooting

| Symptom | Likely cause | Check |
| --- | --- | --- |
| Login fails after register | User still `PENDING_EMAIL_VERIFICATION` | Run the DB activation command or complete email verification. |
| Dataset approval returns `422` | No active test cases | `GET /api/v1/datasets/{datasetPublicId}/test-cases`. |
| Evaluation create returns `422` | Dataset/rubric/connector/judge model is not in required state | Dataset must be `APPROVED`, rubric version `PUBLISHED`, connector active, judge model active. |
| Job stays `PENDING` | Worker disabled or Redis issue | `docker compose logs api`; verify `vqc.worker.enabled=true` and Redis health. |
| CLI job fails before results | Promptfoo binary missing or wrong mode | Check `VQC_PROMPTFOO_MODE`, `VQC_PROMPTFOO_BINARY_PATH`, and `promptfoo --version`. |
| CLI result has no criteria details | Promptfoo output missing `gradingResult.componentResults` | Inspect `runs/{runPublicId}/results.json`. |
| Export file is tiny JSON | Download endpoint returned error | Inspect `GET /api/v1/exports/{exportPublicId}` and job `errorMessage`. |

## Notes For Automation

- `scripts/e2e-smoke.sh` should be updated before it is treated as CI-ready:
  - Use integer criterion weights.
  - Create rubric via `POST /api/v1/rubrics` or patch version `content` before publish.
  - Include `judgeModelPublicId` in evaluation creation.
  - Read `runPublicId` and `exportPublicId` response fields.
- Keep API smoke data unique by timestamping user emails and project names.
- Do not commit generated `runs/`, `exports/`, logs, or downloaded E2E files.
